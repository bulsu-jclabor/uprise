## Architecture Overview

### Two entry points, strict platform split

| Entry point | Target | Roles served |
|---|---|---|
| `lib/main.dart` | Mobile (Android/iOS) | Student, Guest |
| `lib/main_web.dart` | Web (Chrome) | Admin, Org Officers |

`main_web.dart` also intercepts `?verify=CODE` query params to show the public `CertificateVerifyScreen` before any auth.

Role routing is handled by **two separate routers**:
- `lib/role_router.dart` — used by `main.dart`; guards students to mobile only, admins/orgs to web only
- `lib/main_web.dart` `AuthGate` — same logic, used by the web entry

Both routers read the `role` field from the `users` Firestore collection and cache it in `AuthService._roleCache`.

### Two AuthService files

| File | Used by |
|---|---|
| `lib/auth_service.dart` | Web entry (`main_web.dart`, admin/org screens) |
| `lib/services/auth_service.dart` | Mobile screens |

Do not mix these up. The web `AuthService` has a static role cache; the mobile one does not.

### Student data — two Firestore collections

Students exist in **both**:
- `students` — admin-managed, holds `tempPassword`, `course`, `yearLevel`, `orgId`, `mustChangePassword`
- `users` — auth/role data, holds `role`, `orgId`, `mustChangePassword`

When creating or updating a student, write to **both** collections in a batch. The `students` collection UID field is `uid`; the `users` doc ID **is** the UID.

## Key Services

**`lib/services/notification_service.dart`** — `NotificationService`
- `sendToUser(userId, title, body, type)` — single notification
- `sendToOrgMembers(orgId, title, body, type)` — batch to all users with matching `orgId`
- `sendEventNotification(eventId, orgId, title, body)` — to event attendees sub-collection

**`lib/services/activity_logger.dart`** — `ActivityLogger.log(action, module, details)`
Call on every destructive or significant action. Always include `orgId` in `details` for org-scoped actions.

## Export Utilities (platform stubs)

Both admin and org have mirrored export utilities using conditional exports:

```
export_util.dart          →  stub / web / io   (AdminExportUtil.saveBytes / saveText)
export_pdf.dart           →  AdminExportPdf.generateTablePdf(title, headers, rows)
```

The pattern `export 'stub.dart' if (dart.library.html) 'web.dart' if (dart.library.io) 'io.dart'` is used for any file that touches `dart:html` or `dart:io` directly. Do not import `dart:html` directly in shared code — wrap it in the stub/web/io pattern.

## Design Token Pattern

Every major screen file defines its own local color and spacing classes rather than importing from a shared theme:

```dart
class UpriseColors { static const Color primaryDark = Color(0xFFB45309); ... }
class _DS { static const double radiusMd = 12; static final cardShadow = [...]; }
```

`UpriseColors.primaryDark` = `Color(0xFFB45309)` (amber-700) is the brand primary across all files. The accent is `Color(0xFFF59E0B)`. Do not use `withOpacity()` — use `.withAlpha(n)` (0–255 integer) instead.

Admin screens use `AdminColors` (where defined separately), org screens use `OrgColors`. Check the file's local class before referencing a color.

## Certificate Verification Flow (Phase 3)

When a student is marked present via QR scan in `org_attendance_qr.dart`, a certificate is auto-written to `certificates` with:
- `verificationCode`: 12-char alphanumeric
- `autoGenerated: true`
- `recipientId` + `recipientUid`: both set to the student UID (the student screen queries by `recipientUid`)

Public verification is at `/?verify=CODE` — intercepted in `main_web.dart` before auth, renders `lib/screens/public/certificate_verify_screen.dart`.
